The three-layer architecture

The system is intentionally split across three layers. Each one does what the other two cannot.

1 GitHub Actions — Enforcement
Blocks the merge. Deterministic path guard + canary smoke test + Bob headless 10-point audit. Knows: git diff, branch protection, canary token. Runs before any code lands on main.
Can't: reason about violation content, route to multiple systems, answer questions.
2 watsonx Orchestrate — Orchestration Brain
Receives the raw CI JSON payload, decides which tools to call and how many times (one Issue call per violation), inspects return values for failures, applies fallback routing, and answers ad-hoc governance questions in plain English. Serves every repo from one agent.
Can't: own the authoritative GRC record, fire audit-logged email, enforce retention.
3 OpenPages — Authoritative Record
One Issue per violation. Dedup on repo + pr_number + rule. Auto-close on PASS. Native Workflow Engine fires email — audited, throttled, retried by the GRC engine. Trend metrics per repo/actor/rule. NERC CIP control mapping (CIP-007 R5, CIP-010 R1).
Can't: receive a raw CI JSON blob, apply routing logic, answer conversational questions.
wxO is not just a connector. It is the reasoning layer between an unstructured CI event and a structured GRC system. Semantic routing, cross-repo fan-out, fallback orchestration, and conversational Q&A are all wxO responsibilities that neither GitHub Actions nor OpenPages can perform.

4. Configure the Workflow Engine email template

In the OpenPages Workflow Designer, on the AI-AIDLC-Violation Issue type:

  1. Add a notification action on the Open stage transition
  2. Set recipients: your security/CISO distribution list
  3. Use dynamic fields in the template: {{op-iss:Name}}, {{AIDLC_PRUrl}}, {{op-iss:RiskRating}}, {{AIDLC_ControlID}}
  4. Optionally add a notification on the Closed transition for PASS auto-close events

Once this is configured, send_outlook_email becomes the fallback-only path — the Workflow Engine owns all normal email.

Migration path — from fallback email to OpenPages-native email

Create OpenPages schema objects and custom fields
Two object types (AI-AIDLC-Violation, AI-AIDLC-Metrics) and the custom fields listed above. Verify with a manual REST API call before proceeding.
Configure Workflow Engine email template on AI-AIDLC-Violation
Add notification on the Open stage. Use dynamic field references. Test by manually creating a test Issue and confirming email arrives.
Deploy the governance agent with all 5 tools
orchestrate env activate <env> && bash deploy.sh from the aidlc-governance-agent repo. Set all OPENPAGES_* env vars. At this stage OUTLOOK_* are still set as the fallback.
Run a test PR and confirm OpenPages Issue is created and email arrives via OpenPages
Check the agent conversation log — it will state "email sent via OpenPages native workflow" vs "fallback used". Confirm the email was received and the OpenPages audit log shows the notification.
Remove send_outlook_email from the agent and redeploy
In agents/governance-agent.yaml: remove send_outlook_email from the tools: list and the email routing instructions. Remove OUTLOOK_* secrets. Redeploy: bash deploy.sh.
Keep the fallback as long as OpenPages email is not fully verified. The cost is five extra env vars. The benefit is that a misconfigured Workflow Engine template does not silently drop alerts.

Operational notes

All three notification paths are fail-open

Every tool in the governance agent swallows exceptions and returns an error string rather than raising. The notify-wxo.py step in the CI workflow also catches all exceptions. An OpenPages outage, an Outlook outage, or a wxO outage cannot block a merge.

Dedup key — why it matters

The dedup key repo#pr_number#rule means:

  • Re-running the gate on the same PR (e.g. after a force-push) updates the existing Issue rather than creating duplicates
  • When the PR is fixed and re-runs as PASS, the same key is used to find and auto-close the Issue
  • The same PR violating the same rule on two repos creates two separate Issues (different repo prefix)

Why governance changes go directly to main (not via PR)

The deterministic guard blocks any PR that touches .bob/, .github/, governance/, or .bobignore. This is intentional — governance files are reviewed by the code owner and pushed directly to main. A PR cannot carry a governance change alongside a code change because the gate would catch the governance touch and fail the PR before the code is even audited.

The rulebook is pinned from main. The CI workflow restores .bob/rules/ and governance/ from origin/main before running. A PR cannot hand the gate a weakened Constitution to rate itself against.
