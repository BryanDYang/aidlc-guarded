The three-layer architecture

The system is split across three layers. Each one does what the other two cannot.

1 GitHub Actions — Enforcement
Blocks the merge. Deterministic path guard + canary smoke test + Bob headless 10-point audit. Knows: git diff, branch protection, canary token. Runs before any code lands on main.
Can't: reason about violation content, route to multiple systems, answer questions.
2 watsonx Orchestrate — Orchestration Brain
Receives raw CI JSON, decides which tools to call and how many times (one Issue call per violation), inspects return values for failures, applies fallback routing, answers ad-hoc governance questions in plain English. One agent serves every repo.
Can't: own the authoritative GRC record, fire audit-logged email, enforce retention.
3 OpenPages — Authoritative Record
One Issue per violation. Dedup on repo + pr_number + rule. Auto-close on PASS. Native Workflow Engine fires email — audited, throttled, retried by the GRC engine. Trend metrics per repo/actor/rule. NERC CIP control mapping (CIP-007 R5, CIP-010 R1).
Can't: receive a raw CI JSON blob, apply routing logic, answer conversational questions.
wxO is not just a connector. It is the reasoning layer between an unstructured CI event and a structured GRC system. Semantic routing, cross-repo fan-out, fallback orchestration, and conversational Q&A are all wxO responsibilities that neither GitHub Actions nor OpenPages can perform.

Operational notes

All notification paths are fail-open

Every tool swallows exceptions and returns an error string. The notify-wxo.py CI step also catches all exceptions. An OpenPages, Outlook, or wxO outage cannot block a merge.

Dedup key

The dedup key repo#pr_number#rule means re-running the gate on the same PR updates the existing Issue rather than creating duplicates. When the PR is fixed and passes, the same key finds and auto-closes the Issue.

Why governance changes go directly to main

The deterministic guard blocks any PR touching .bob/, .github/, governance/, or .bobignore. Governance files are reviewed by the code owner and pushed directly to main. The rulebook is pinned from main before each audit — a PR cannot hand the gate a weakened Constitution to rate itself against.
