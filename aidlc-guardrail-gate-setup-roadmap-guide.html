Part 1 — GitHub Action Setup

Everything is already wired in .github/workflows/guardrail-gate.yml. You only need to configure three things in your GitHub repository settings:

Step-by-step

Add the BOBSHELL_API_KEY repository secret
The workflow installs Bob Shell via curl -fsSL https://bob.ibm.com/download/bobshell.sh | bash and then authenticates headlessly with --auth-method api-key. Without this key the audit step errors.

Go to Settings → Secrets and variables → Actions → New repository secret:
Name: BOBSHELL_API_KEY · Value: your IBM Bob API key.
The key is used by bob --auth-method api-key inside the runner. It never appears in logs.
Enable branch protection on main
Go to Settings → Branches → Add branch ruleset (or classic protection) for main:

  • ☑ Require a pull request before merging
  • ☑ Require status checks to pass — add check: bob-guardrail-audit
  • ☑ Do not allow bypassing the above settings
  • ☑ Restrict who can push to matching branches (admins only for direct push)
The check name must match exactly — it is the name: field inside the bob-audit job in the YAML: bob-guardrail-audit.
Enable GitHub Pages on the gh-pages branch
The dashboard step clones / creates a gh-pages branch and pushes index.html + data/violations.json there after every PR run.

Go to Settings → Pages:
Source: Deploy from a branch · Branch: gh-pages · Folder: / (root)
The workflow uses GITHUB_TOKEN (auto-provided) to push. No extra secret needed for this step — the job already requests contents: write.
Confirm CODEOWNERS is respected
.github/CODEOWNERS maps /.bob/, /.bobignore, /.github/, /governance/, and /dashboard/ to @BryanDYang. Any PR touching those paths needs an approving review from that owner before it can merge — even if the bot audit passes.
Change the username to the right code-owner for your org before going to production.

Secret / variable reference

What runs on every PR

PR opened against main
▼
Pin rulebook from main (governance can't self-certify)
▼
Deterministic guard — did the PR touch .bob/, governance/, .github/, .bobignore?
▼
Canary smoke test — did Bob actually load the workspace rules?
▼
Bob headless audit (advanced, read-only) — 10-point review of the diff
▼
PASS → green, mergeable
FAIL → blocked + reason posted as PR comment
▼
Dashboard record pushed to gh-pages
wxO governance agent notified (if secrets set)
Key design property: the last two steps (dashboard and wxO notify) are both if: always() + continue-on-error: true — they never block the gate. A governance or wxO outage cannot freeze the trunk.

Part 2 — watsonx Orchestrate (wxO)

The agent is built (aidlc-governance-agent repo). The CI workflow already calls governance/notify-wxo.py on every run — it skips silently if the two secrets aren't set. To activate:

① Deploy the agent into your wxO environment

The agent has 5 tools: record_pr_verdict, query_verdict_dashboard, create_or_update_openpages_issue, update_openpages_metrics, send_outlook_email (fallback only).

cd aidlc-governance-agent
orchestrate env activate <your-env>
bash deploy.sh

Then get the agent's chat URL: orchestrate agents list --verbose — copy the chat_url.

② Set the two GitHub secrets
  • WXO_API_KEY — IBM Cloud API key, exchanged for a short-lived IAM bearer token at runtime
  • WXO_AGENT_URL — the chat-completions URL from the step above

Once both are set the next PR run will automatically send governance events to the agent.

③ How the token exchange works

notify-wxo.py already implements this pattern — no changes needed:

# 1. Exchange IBM Cloud API key → short-lived IAM bearer token
POST https://iam.cloud.ibm.com/identity/token
  grant_type=urn:ibm:params:oauth:grant-type:apikey
  apikey=$WXO_API_KEY

# 2. POST the governance event to the wxO agent chat endpoint
POST $WXO_AGENT_URL
Authorization: Bearer <iam_token>
{
  "stream": false,
  "messages": [{"role": "user", "content": "<event_json>"}]
}
④ Governance event schema

Every event the gate sends follows aidlc-governance-event/v1:

{
  "schema":     "aidlc-governance-event/v1",
  "source":     "github-actions",
  "repo":       "org/repo-name",
  "project":    "UTILITY-AIDLC",
  "actor":      "dev-username",
  "pr_number":  "42",
  "verdict":    "FAIL",
  "severity":   "critical",
  "violation_count": 2,
  "violations": [
    { "file": "api/customers.py",
      "rule": "01-secure-coding.md",
      "detail": "Hardcoded API key...",
      "severity": "critical" }
  ],
  "run_url":    "https://github.com/.../actions/runs/...",
  "timestamp":  "2025-07-15T14:23:00Z"
}

Severity is auto-classified by notify-wxo.py using rule / detail / file text keywords — same logic as the dashboard.

Part 3 — OpenPages Integration

OpenPages is the authoritative GRC record. The wxO agent creates one Issue per violation via REST API. OpenPages' Workflow Engine owns email in production — the wxO send_outlook_email tool is a fallback only, firing when OpenPages is unreachable.

① Create schema objects in OpenPages
  • Object type AI-AIDLC-Violation — one record per rule violation, dedup key repo#pr_number#rule, auto-close on PASS
  • Object type AI-AIDLC-Metrics — one record per repo, incremental pass/fail counters + per-rule/per-actor tallies
  • Custom fields: AIDLC_DedupKey, AIDLC_Repo, AIDLC_Project, AIDLC_PRUrl, AIDLC_ControlID, AIDLC_ReportedBy

See the full field list in aidlc-guardrail-gate-prod-considerations.html.

② Configure Workflow Engine email on AI-AIDLC-Violation

In the OpenPages Workflow Designer, add a notification action on the Open stage transition. Once this is working and verified, remove send_outlook_email from the agent tools list and retire the OUTLOOK_* secrets. The production email path is then fully owned by OpenPages — audited, throttled, and retried by the GRC engine.

③ Trend metrics and reporting
  • Violations by rule over time · violations by actor · pass rate by repo/project
  • Mean time to remediation per severity tier
  • NERC CIP control mapping — CIP-007 R5, CIP-010 R1
